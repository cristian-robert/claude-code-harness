# Does `.codex/agents/*.toml` accept a `model` key?

**Verdict: ACCEPTED** — `model` and `model_reasoning_effort` do not trip a schema error
in `.codex/agents/*.toml`. Confidence comes mostly from static evidence (see "Why", below);
the live tests are consistent with acceptance but turned out not to be as conclusive as
they first looked (see "What the live tests do and don't prove").

## What the live tests do and don't prove (important caveat)

Before trusting Step 2 at face value, I ran a differential control: I appended an
obviously bogus key (`totally_bogus_field_xyz = "should-not-exist"`) to a *different*
agent file (`scout.toml`) and re-ran the same `codex exec` prompt. It **also** produced
zero errors, exit code 0. That result would be surprising if the file were being
strictly validated — an outright made-up field name should be the easiest possible
rejection. This means the initial "no error" from Step 2 alone under-determines the
question: it is also consistent with `codex exec` simply never touching
`.codex/agents/*.toml` when nothing in the session actually dispatches a subagent.

I then found (see "Why", below) that `.codex/config.toml` never actually *registers*
these agent files anywhere — there is no directory auto-scan of `.codex/agents/` in the
binary. So I built the real registration Codex expects and reran:

```toml
# appended to $SCRATCH/.codex/config.toml
[agents.code-reviewer]
description = "Reviews a diff against its plan and the repo's rules."
config_file = ".codex/agents/code-reviewer.toml"
nickname_candidates = ["code-reviewer"]
```

```bash
cd "$SCRATCH" && codex exec --skip-git-repo-check -m gpt-5.6-luna \
  'Reply with exactly: SCHEMA_OK' < /dev/null
```

Result: exit 0, clean `SCHEMA_OK`, same as before — this time with the `model` /
`model_reasoning_effort` file actually registered as a real agent role.

To rule out "registration exists but content is still never parsed," I ran a negative
control: removed the **required** `name` key from the same (now-registered)
`code-reviewer.toml` and reran the identical command. This should, per the binary's own
error strings (below), fail with `agent role file at .. must define a non-empty
'name'`. It did not — exit 0, clean `SCHEMA_OK` again.

**Conclusion from the live tests alone:** a plain `codex exec '<prompt>'` invocation
that never causes the model to actually spawn/dispatch a subagent does not appear to
touch `.codex/agents/*.toml` content at all — registration or not, valid or broken, the
result is identical. So neither the positive result (model key present, no error) nor
the negative control (required field missing, no error) is, by itself, proof of
anything about the schema. I did not find a CLI flag to force an explicit subagent
spawn (`codex exec --help` / `codex --help` have no such option), and forcing the model
to voluntarily invoke its own spawn tool inside a 90s non-interactive run was not
reliable, so I did not chase that further — see "Why" below for the evidence that
substitutes for it.

## Why (static evidence from the binary) — this is what the verdict actually rests on

`strings` + a raw byte-offset scan of the codex binary
(`codex-darwin-arm64/vendor/aarch64-apple-darwin/bin/codex`) turned up the real
mechanism, in `core/src/config/agent_roles.rs`:

- Agents are **registered** in `config.toml` under `[agents.<name>]`, deserialized as
  `codex_config::config_toml::AgentRoleToml` (confirmed via the mangled Rust symbol
  `_...codex_config..config_toml..AgentRoleToml...deserialize...`). This struct has 3
  fields: `description`, `config_file`, `nickname_candidates`. Error strings confirm:
  `"agents.<name>.config_file must point to a file"`,
  `"agents.<name>.config_file must point to an existing file at <path>"`,
  `"duplicate agent role name '<name>' declared in config"`.
  **There is no directory auto-scan of `.codex/agents/` anywhere in the binary** — a
  plain `strings` search for `.codex/agents` and `codex/agents` returns zero hits.
  PHE's `.codex/config.toml` currently emits no `[agents.<name>]` blocks at all, so
  none of PHE's four generated agent files are wired to anything today, independent of
  the `model` key question.

- `config_file` points at the actual per-agent TOML — the "agent role file" — which
  deserializes into `RawAgentRoleFileToml`. Critically, the derived serde error
  message for this type is **`"a map"`** (`struct RawAgentRoleFileTomla map` in the
  string table), not `"struct RawAgentRoleFileToml"` the way every other
  `#[derive(Deserialize)]` struct in the binary presents (compare: `"struct
  AgentRoleToml with 3 elements"`, `"struct ConfigToml with 97 elements"`, etc. — all
  named-field structs report as `struct <Name> with N elements`). `"expecting: a map"`
  is serde's default `Visitor::expecting()` text for types deserialized generically as
  a table (`toml::Table` / `HashMap<String, toml::Value>`), which is structurally
  **incompatible with `#[serde(deny_unknown_fields)]`** — that attribute only has
  meaning for a fixed-field struct. A map accepts any key by construction.

- Consistent with that, the required-field checks for the role file are clearly
  hand-written validation over the parsed map, not serde's automatic field errors —
  the messages read as prose, not serde's mechanical `missing field '<x>'` /
  `unknown field '<x>', expected 'a', 'b', 'c'` template (which does appear elsewhere
  in the binary, e.g. for `ConfigToml`, confirming the binary *does* use
  `deny_unknown_fields` in general — just not here):

  ```
  agent role file at .. must contain a TOML table
  failed to parse agent role file at ..: ..
  failed to deserialize agent role file at ..: ..
  agent role file at .. must define a non-empty `name`
  ... cannot be blank
  ... must contain at least one name        (nickname_candidates)
  ... cannot contain blank names
  ... cannot contain duplicates
  ... may only contain ASCII letters, digits, spaces, hyphens, and underscores
  agent role `..` must define a description
  agent role file at .. must define `developer_instructions`
  agent role file at .. developer_instructions cannot be blank
  ```

  I searched exhaustively for an "unknown key" / "unrecognized key" / "unsupported
  key" style error anywhere near this string cluster (`agent_roles.rs`) and found
  none — the only "unknown field ..." and "unexpected keys in table: .., available
  keys: .." strings in the whole binary belong to serde's generic derive machinery and
  the `toml` crate's own parser errors, used elsewhere (e.g. `ConfigToml`, which is
  confirmed elsewhere to use `deny_unknown_fields`), not to this code path.

Put together: the file `model` and `model_reasoning_effort` would land in is parsed as
a generic table, the validation code only ever asks the table for `name`,
`description`, `developer_instructions` (and possibly `nickname_candidates`), and
nothing in the binary enumerates or rejects unrecognized keys in that table. There is
no mechanism by which an extra key could produce a hard error.

## Verdict

**ACCEPTED.** `model` and `model_reasoning_effort` do not cause a schema/parse error in
`.codex/agents/*.toml`. This is well-supported by static analysis of the binary
(`RawAgentRoleFileToml` is a generic map, structurally unable to enforce
`deny_unknown_fields`) and is not contradicted by either live test — it just isn't
strongly confirmed by them either, because a plain `codex exec` prompt never appears to
trigger actual parsing of agent role files (see caveat above). I could not find a CLI
path to force a real subagent spawn non-interactively within the time/attempt budget
this task allows, so treat the "live" portion of this verdict as corroborating, not
load-bearing — the load-bearing evidence is the binary's own struct/error-string
analysis.

Per the brief's Step 4: **ACCEPTED → no code change.** I have not modified
`cli/emit-codex.js` or anything else.

## Separate finding to flag for the controller — **RESOLVED in `ce691d3`**

> **Status update (amended 2026-07-12).** The finding below was true when written and is now **fixed**.
> Commit `ce691d3` ("feat(codex): resolve agent tier -> model + effort, and register the agents in
> config.toml") taught `cli/emit-codex.js` to emit a `[agents.<name>]` block per agent — with
> `description`, `config_file`, and `nickname_candidates` — so the generated agent files are now
> registered and reachable. The original text is preserved verbatim below as the evidence trail that
> found it; read it as a record of the bug, not of current behavior.

Regardless of the `model` key question: **PHE's generated `.codex/config.toml` never
registers any of the four `.codex/agents/*.toml` files it generates.** There is no
directory auto-discovery in Codex — an agent only exists to Codex if `config.toml` has
a matching `[agents.<name>]` block with `description`, `config_file` (pointing at the
`.toml` file), and `nickname_candidates`. Today's `CODEX_CONFIG_TOML` constant in
`cli/emit-codex.js` only emits `[features] multi_agent = true` and `[agents]
max_threads = 4` — no per-agent blocks. So as things stand, none of
`architect-agent`, `code-reviewer`, `qa-evaluator`, `research-gatherer`, or `scout` are
actually reachable through Codex's multi-agent system, independent of whatever this
plan does with `model`. This looks like it belongs in whatever task wires the model
tier map into the Codex emitter (it needs `[agents.<name>]` registration blocks
regardless), but I'm flagging it rather than fixing it since it's outside this task's
file scope.
